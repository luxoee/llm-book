# 商业化发布校验清单

这份清单用于本书发布到 GitBook、知识付费平台或企业内训前的最后一轮深度校验。

## 1. 市场定位

- [ ] 书名是否明确传达“LLM 推理系统 / vLLM / Qwen3.6 / GPU serving”。
- [ ] 首页是否说明读者能解决的真实问题，而不是只列技术名词。
- [ ] 是否明确适合读者和不适合读者。
- [ ] 是否说明本书与普通 Transformer 教程、部署命令合集、源码碎片文章的差异。
- [ ] 是否形成可销售的学习收益：能读模型卡、能估显存、能理解调度、能定位瓶颈、能规划部署。

## 2. 目录与内容分配

- [ ] 第 1 章是否承担导读、事实边界和路线图，而不只是校验清单。
- [ ] 第 2 章是否建立传统 decoder-only baseline。
- [ ] 第 3–5 章是否把 Qwen3.6 的模型计算链路讲清楚。
- [ ] 第 6–8 章是否形成显存、KV cache、PagedAttention、调度的连续叙事。
- [ ] 第 9–11 章是否形成 MoE、并行、GPU kernel 的性能叙事。
- [ ] 第 12–14 章是否能支撑源码阅读、部署上线、压测排障。
- [ ] 第 15 章是否给出可执行学习路线，而不只是重复总结。

## 3. 技术事实

- [ ] 所有 Qwen3.6 模型结构、参数量、上下文长度、专家数量、MTP、FP8 相关结论都绑定模型卡或官方来源。
- [ ] 所有 vLLM 版本、源码路径、部署命令都固定在 v0.20.1 或明确标注来源。
- [ ] 未读源码的结论是否标为“待源码确认”或“合理推断”。
- [ ] 是否避免把 Qwen3.6 简化成普通 Transformer Decoder。
- [ ] 是否避免把 FP8 checkpoint 误写成“显存问题全部解决”。
- [ ] 是否避免把源码目录存在等同于运行时一定选中该 backend。

## 4. 工程可执行性

- [ ] 部署命令是否区分标准模式、text-only、MTP、FP8、BF16 多卡、Expert Parallel。
- [ ] 实验作业是否说明观测指标，例如 TTFT、TPOT、tokens/s、KV cache usage、GPU memory、NCCL 通信。
- [ ] 排障章节是否覆盖启动 OOM、prefill OOM、decode OOM、MoE imbalance、NCCL、backend fallback。
- [ ] 是否明确哪些实验需要真实 GPU 环境，哪些只能作为阅读作业。

## 5. GitBook 发布质量

- [ ] 根目录存在 `README.md`。
- [ ] 根目录存在 `SUMMARY.md`。
- [ ] `SUMMARY.md` 中所有链接都存在。
- [ ] 每个章节只有一个非代码块 H1。
- [ ] 章节内标题层级一致。
- [ ] 代码块 fence 成对闭合。
- [ ] 表格在 GitBook 中可读。
- [ ] 没有 `Prompt`、`Response`、`Thought for`、浏览过程等聊天导出痕迹。
- [ ] `raw/` 原始材料不会被当作正式章节入口。

## 6. 商业化发布前人工复核

- [ ] 请至少找一位懂 LLM serving 的技术审稿人检查第 6–14 章。
- [ ] 请至少找一位目标读者阅读第 1、2、13、14 章，确认是否有购买价值。
- [ ] 请抽样跑通至少一组部署或压测命令，记录硬件环境和结果。
- [ ] 请确认所有外部引用链接仍可访问。
- [ ] 请确认书中涉及的版本边界在销售页和正文中一致。
